# 翻訳リソースの構成ガイド

文言（i18n キー）をどこに置くかの**唯一のルール**。新しい文言は必ずこの手順で配置する。

## 原則: namespace は「サーフェス（プロダクト上の面）」で切る

ファイルの場所でもコンポーネント単位でもなく、その文言が **どの面に属するか** で決める。namespace は次の **2 種類だけ**:

- **`common`** — アプリシェル。ルートレイアウトが描画し **全ページに出る** もの。
  - 対象: サイト識別（`siteName`）/ `<head>` メタ（`metadata`）/ グローバルナビ（`nav`）/ フッター（`footer`）/ テーマ等のグローバル操作（`themeToggle`）。
  - **「共有文言の置き場」ではない。** シェルに描画されないものは入れない（この厳格さが肥大化を防ぐ）。
- **`<feature>`**（`home`, `blog`, …）— 1 フィーチャ領域につき 1 つ。
  - そのフィーチャのページも、**フィーチャ専用コンポーネントの文言も**すべてここに入れる（「コンポーネントだから別 namespace」にはしない。文言は所属する面に従う）。

### やらないこと

> ❌ `ui` / `shared` / `misc` のような「横断汎用」namespace は作らない。
>
> membership ルールが無いため必ずゴミ箱化する。汎用に見えるラベルでも文脈で訳が変わるのが普通なので、 **各フィーチャが自前で持つ（textual な重複は許容）**。本当にグローバル chrome になったときだけ `common` へ昇格する。

## 新しい文言をどこに置くか（決定手順）

1. 全ページ共通のシェル（header / nav / footer / グローバル操作 / メタ）に出るか? → **`common`**（該当する領域グループへ）
2. それ以外 → **その文言を使う唯一のフィーチャ**の namespace へ。複数フィーチャで似た文言が要るなら **各々が持つ**（共有バケツは作らない）。

## キー命名

- 要素名ではなく **概念 / 役割** で付ける。
- 第 2 階層は「面の中の領域 / 役割」でグルーピングする（例: `common` の `metadata` / `nav` / `footer` / `themeToggle`、`blog` の `list` / `post`）。
- 可変値は `{{var}}`、複数形は `key_one` / `key_other`（i18next 標準）を使う。**訳文の文字列連結は禁止**。

## 追加手順

- **文言追加**: 上の決定手順で namespace を選び JSON にキーを足す。`t()` で即型補完される（タイポ / リネーム漏れは `next build` の tsc で落ちる）。
- **フィーチャ追加**: `locales/<lng>/<feature>.json` を作り、`../resources.ts` に import を 1 行足す。型・namespace 一覧は自動追従。`getT("<feature>")` / `useTranslation("<feature>")` で参照。
- **ロケール追加**: `locales/<lng>/` を ja と同じ namespace 構成で複製する。ja がキーの **正本** として型検証されるため、他ロケールのキー過不足は検出できる。

## 現状の namespace

| namespace | サーフェス | 主なキー |
| --- | --- | --- |
| `common` | アプリシェル（全ページ） | `siteName`, `metadata.*`, `nav.*`, `footer.*`, `themeToggle.*` |
| `home` | トップページ | `lead`, `recentPosts` |
| `blog` | ブログ（一覧 + 記事） | `title` |
